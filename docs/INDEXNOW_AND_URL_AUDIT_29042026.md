## 1. Что такое IndexNow

**IndexNow** — это open-source протокол быстрого индексирования, разработанный Microsoft и Yandex в 2021 году. На 2026 год его поддерживают:

- ✅ **Bing** (Microsoft)
- ✅ **Yandex** ⭐ (наш главный канал!)
- ✅ Naver (Корея)
- ✅ Seznam (Чехия)
- ✅ Yep (DuckDuckGo-related)

❌ **Google не поддерживает** (тестировали с 2021, но не приняли)

### Как работает

Вместо того чтобы **ждать** когда поисковый бот придёт на сайт (обычно 1-7 дней), сайт **сам уведомляет** поисковик о новой/изменённой странице через простой HTTP POST.

```
Старый flow: Опубликовали → ждём 3-7 дней → бот зашёл → проиндексировал
Новый flow: Опубликовали → POST в /indexnow → через 5-30 минут страница в индексе
```

### Зачем нам IndexNow

**Текущая ситуация:** мы используем **Yandex Webmaster Recrawl API** — лимит **910 URL/день**, скорость 5-15 минут на обход. Это уже работает.

**IndexNow даёт:**
1. **Нет лимита на дневные отправки** (только rate-limit на запросы)
2. **Один POST = уведомление в Bing + Yandex + Naver одновременно**
3. **Автоматическая отправка при каждом UPDATE в БД** — без ручного recrawl-cron
4. **Можно отправлять URL удалённых страниц** (для очистки индекса) — Yandex Webmaster это не умеет

#### Вариант B — со стороны Claude SEO pipeline

Я добавляю функцию в `yandex_recrawl.py`:

```python
def notify_indexnow(urls: list[str]) -> dict:
    payload = {
        "host": "gaming-goods.ru",
        "key": INDEXNOW_KEY,
        "keyLocation": f"https://gaming-goods.ru/{INDEXNOW_KEY}.txt",
        "urlList": urls[:10000]  # max per batch
    }
    r = requests.post("https://yandex.com/indexnow", json=payload, timeout=15)
    return {"http": r.status_code, "body": r.text[:200]}
```

Каждый раз после массовой генерации SEO — отправляем все URL одним пакетом. Это **дополнительно** к Yandex Webmaster (страховка).

## 4. ⚠️ Идеальны ли наши CS-URL? Нет, не все.

Из **39 CS-страниц** проверил структуру URL — **5 имеют дефекты** (13%):

### 🔴 URL-проблемы

| URL | Проблема | Что должно быть |
|-----|----------|------------------|
| `/t/g4skins-com-gift-card` | `-com-` в slug | `/t/g4skins-gift-card` |
| `/t/pirateswap-gift-card-gift-card` | **дубль** «gift-card-gift-card» | `/t/pirateswap-gift-card` |
| `/t/rainbet-gift-card-gift-card` | **дубль** «gift-card-gift-card» | `/t/rainbet-gift-card` |
| `/t/daddyskins-other` | категория «other» вместо gift-card | `/t/daddyskins-gift-card` |
| `/t/bountystars-com-gift-card` | `-com-` в slug | `/t/bountystars-gift-card` |

### Остальные 34 URL (87%) — **идеальны** ✅

- `/t/hellcase-gift-card`
- `/t/farmskins-game-keys`  
- `/t/skin-club-gift-card`
- `/t/rain-gg-gift-card`
- ... и так далее

### Почему это плохо для SEO

**Дубли «gift-card-gift-card»** = поисковик видит:
- Подозрительную URL-структуру (auto-generated артефакт)
- Дубль ключевого слова — **ключевое слово stuffing** (Яндекс может понизить)
- Возможна подозрение на «cloaking» / спам

**`-com-` в slug** = тоже неестественно. Slug должен описывать **содержимое страницы**, а не быть транслитерацией доменного имени.

**`-other`** = категория «прочее» — не имеет SEO-смысла. Поисковик не поймёт что страница про gift-card.

### Как исправить — это **задача Саше** (slug-маппинг)

Этот баг известен ещё с задачи `TASK_SASHA_PAYPAL_AGGREGATOR_21042026.md` — slug-генератор автоматически создаёт URL'ы с дублями `gift-gift-gift-card` для агрегаторов. Саша это знает.

### План действий

1. **Саша:** правит slug-маппинг → новые URL без дублей будут:
   - `/t/pirateswap-gift-card` (вместо `pirateswap-gift-card-gift-card`)
   - `/t/rainbet-gift-card`
   - `/t/g4skins-gift-card`
   - `/t/daddyskins-gift-card` (создать новый, переадресовать со старого)
   - `/t/bountystars-gift-card`

2. **Backend:** автоматический 301 редирект со старых на новые
3. **Claude:** перенесу SEO с старых URL на новые (один UPDATE в БД)
4. **Через IndexNow** уведомляю Yandex/Bing об удалении старых URL и добавлении новых
5. Через 1-2 недели старые URL вылетают из индекса, новые занимают их место
