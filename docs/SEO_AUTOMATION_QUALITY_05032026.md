# Как гарантировать качество n8n SEO-автоматизации
**gaming-goods.ru | 05.03.2026**

---

## Проблема: диагноз vs автопилот

Сегодня за 30 минут ручной диагностики найдено:
1. 113 265 страниц без `<title>`
2. Каннибализация `/t/nutaku-currency` vs `/marketplace/nutaku`
3. Третий URL-формат `/trade/` о котором никто не думал
4. CTR 1.3% на позиции 7 — аномалия требующая внимания

Вопрос правильный: **сможет ли WF-202 находить такие вещи автоматически?**

Честный ответ: **нет, если строить его так как сейчас.**

---

## Почему WF-202 в текущем виде — это не "SEO специалист"

WF-202 делает одно: берёт 25 запросов и 20 страниц из GSC → отправляет Клоду → получает 6-8 строк инсайта.

Это как нанять аналитика и давать ему только один отчёт из 10 доступных.

| Что нашла ручная диагностика | Видит ли WF-202 сейчас |
|------------------------------|----------------------|
| 113K страниц без title | ❌ Нет — данные в PostgreSQL |
| Каннибализация URL | ❌ Нет — нужно сравнивать /t/ vs /marketplace/ |
| CTR 1.3% при позиции 7 | ✅ Да — если страница попала в топ-20 |
| Три формата URL | ❌ Нет — нет доступа к page_seo таблице |
| Нет русских title на /t/ | ❌ Нет — нет доступа к marketplace таблице |

## Фреймворк: 4 уровня качества автоматизации

### Уровень 0 — "Отчёт" (текущий WF-202)
Берёт поверхностные данные → Claude пишет вывод.
Качество: **30%** от ручного анализа.
Польза: тренд дня, аномалии в кликах. Не более.

### Уровень 1 — "Наблюдатель" (WF-202 улучшенный)
Добавить в контекст:
- Сравнение /t/ vs /marketplace/ (SQL запрос)
- Страницы с показами > 100 и CTR < 3%
- Динамика vs прошлая неделя (delta)

Качество: **55%** от ручного анализа.
Что теряет: структурные проблемы кода, новые URL форматы.

### Уровень 2 — "Аудитор" (WF-205 еженедельный)
Раз в неделю полный прогон:
- Все 25 топ-страниц + сравнение с /t/ аналогами в БД
- Canonical конфликты (запрос к marketplace + GSC cross-reference)
- Страницы с 0 title в БД (прямой SQL)
- Сравнение с прошлой неделей по всем метрикам

Качество: **80%** от ручного анализа.
Что теряет: неизвестные таблицы (как нашли page_seo сегодня).

### Уровень 3 — "Полный специалист" (цель)
Добавить прямой доступ к БД в агент:
- PostgreSQL Tool node в n8n AI Agent
- Агент сам пишет SQL запросы когда нужно
- Может сам обнаружить page_seo таблицу через information_schema

Качество: **95%** от ручного анализа.

## Конкретный план апгрейда WF-202

### Шаг 1 — Добавить SQL-контекст в Prepare ноду (2 часа)

```javascript
// Добавить к существующему GA4+GSC контексту:

// Запрос 1: каннибализация - страницы есть в обоих форматах
SELECT 
  m.slug_suggestion,
  m.seo_data->'ru'->>'title' as has_title
FROM marketplace m
WHERE m.slug_suggestion IS NOT NULL
  AND m.updated_at > now() - interval '7d'
LIMIT 50

// Запрос 2: сколько /t/ страниц без title (агрегат)
SELECT 
  COUNT(*) as total,
  COUNT(CASE WHEN seo_data->'ru'->>'title' IS NOT NULL THEN 1 END) as with_title,
  COUNT(CASE WHEN seo_data IS NULL OR seo_data = '{}' THEN 1 END) as empty_seo
FROM marketplace
WHERE slug_suggestion IS NOT NULL
```

Это дает Клоду контекст чтобы сказать: "Из 113K страниц только X% имеют title — приоритет №1".

### Шаг 2 — Delta метрики (неделя к неделе)

Текущая проблема: WF-202 каждый день видит одни и те же данные без сравнения.
Клод не знает — стало лучше или хуже.

Решение: хранить снапшот в ai_manager_context таблице (она уже есть в БД):
```sql
-- Сохранять каждую неделю:
INSERT INTO ai_manager_context (context_type, data, created_at)
VALUES ('seo_snapshot', $weekly_metrics, now())
ON CONFLICT (context_type) DO UPDATE SET data = $weekly_metrics
```

Тогда Клод видит: "CTR вырос с 3.2% → 4.1% за неделю. Позиция /t/nutaku-currency: 6.7 → 5.2."

### Шаг 3 — Эталонные проверки (чеклист)

Добавить в промпт Клода жёсткие правила:

```
Каждый анализ ОБЯЗАН проверить:
1. Есть ли страницы с позицией < 10 и CTR < 3%? → Canonical или title проблема
2. Есть ли запросы с > 500 показов и 0 кликов? → Страница не проиндексирована
3. Ratio /t/ vs /marketplace/ в топе → Каннибализация
4. Недельная динамика avg position → Тренд
```

## Что нельзя автоматизировать (честно)

| Находка | Почему автомат не найдёт |
|---------|------------------------|
| `/trade/` третий формат URL | Нужно знать что искать. Автомат не читает код |
| "Что рендерится в `<title>` реально" | Нужен headless browser / rendering check |
| Качество контента на странице | Claude видит только метрики, не страницу |
| Новые баги в деплое | Нет мониторинга фронтенда |

**Для этого нужен живой человек + Claude Friendly миграция** — тогда Claude-разработчик
сможет видеть реальный рендеринг страниц.

## Система контроля качества автоматизации

### Принцип: автоматизация должна знать о себе

WF-202 должен логировать не только инсайты — но и **уверенность в данных**:

```
🤖 SEO агент · 05.03.2026

[ДАННЫЕ]
✅ GA4: 30 дней, 8 306 сессий
✅ GSC: 20 страниц, 364 клика
✅ БД: 113 265 /t/ страниц, 0% с title (КРИТИЧНО)
⚠️ Яндекс Метрика: данные 30 дней (не 24ч)
❌ Indexing API: нет — новые страницы не отправляются

[ИНСАЙТ]
...
```

Если агент не знает о пробеле в данных — он даёт неполный анализ уверенно.
Лучше явная пометка "❌ нет доступа к X" чем тихое игнорирование.

### Еженедельный "health check" автоматизации

Добавить в WF-205 (еженедельный аудит) раздел:

```
📊 HEALTH CHECK АВТОМАТИЗАЦИИ
  WF-201 (отчёт): ✅ 7/7 запусков успешно
  WF-202 (агент):  ✅ 7/7 запусков успешно
  WF-204 (индекс): ❌ 0/7 — не настроен
  БД данные:      ✅ marketplace актуален (last: сегодня)
  GSC данные:     ✅ задержка 2 дня (норма)
```
