## 1. Что из себя представляет пайплайн

### Архитектура (верхний уровень)

```
Очередь в БД (60K /t/ URL)
    ↓
fetch_queue()        — берёт N URL без SEO-контента
    ↓
probe_product()       — проверяет «живая ли страница» 
    ↓                      (ЗДЕСЬ БЫЛА ДЫРА — см. раздел 3)
ALIVE-список (N штук)
    ↓
generate_for_page()   — параллельно 10 запросов в Anthropic API
    ↓
validate()            — проверяет JSON-валидность + длины полей
    ↓
UPDATE page_seo       — запись в БД построчно (10 rows / tx)
    ↓
yandex_recrawl.py     — отправка URL в Yandex Webmaster 
                        (ЗДЕСЬ ТОЖЕ БЫЛА ДЫРА — см. раздел 3)
```

### Ключевые файлы
- `seo_generator.py` — главный пайплайн, ~870 строк
- `seo_prompt.py` — системный и user промпты, post-processing, валидатор
- `pg.py` — обёртка для SQL через n8n webhook
- `gen_log.py` — журнал генерации (стоимость, токены, дедупликация на 90 дней)
- `yandex_recrawl.py` — отправка URL в Яндекс.Вебмастер API

### Модель
- **Основная:** `claude-sonnet-4-5-20250929` (Sonnet 4.6) — $3/1M input, $15/1M output
- **Fallback:** `claude-opus-4-7` (Opus 4.7) — если Sonnet валидатор 3 раза отверг
- **Параметры:** `max_tokens=1800`, `temperature=0.3`, `parallelism=10`

### Стоимость
- Средняя: **$0.03 / страница** (в основном Sonnet)
- Opus fallback: $0.12-0.18 / страница (редко)
- Dev cost за неделю: **$85** (2973 страницы сгенерировано)

## 3. Что пошло не так

### Главный провал: **неверная проверка живости страницы**

В `probe_product()` (seo_generator.py строка 151) была логика:

```python
def probe_product(url: str) -> dict:
    slug = url.rstrip("/").rsplit("/", 1)[-1]
    r = requests.get(f"https://gaming-goods.ru/api/v1/t/{slug}", ...)
    if r.status_code == 404:
        return {"alive": False}   # ← ПРОБЛЕМА
    product = r.json().get("data", {}).get("product")
    if not product:
        return {"alive": False}
    return {"alive": True}
```

**Проблема:** API `/api/v1/t/{slug}` возвращает **404** для агрегаторов у которых slug не совпадает с `brand` в таблице `products` (баг slug-matching на backend).

Но **SSR-рендеринг** (Nuxt) тех же страниц **может показывать товары** — потому что берёт данные из другого источника (прямой SQL-запрос, не через API).

Результат:
- API говорит «404, товаров нет»
- SSR показывает «Товар не найден» (у ~17% URL — действительно пустые)
- SSR показывает **валидный товарный листинг** (у ~83% URL — нормальные страницы)

Мой probe **всегда** помечал все эти URL как dead и **не генерировал** на них SEO (потеря) ИЛИ генерировал когда API временно возвращал какие-то данные (фальшивый alive).

### Второй провал: **отсутствие re-check перед отправкой в Яндекс**

`yandex_recrawl.py` брал из БД все URL где `page_seo.article >= 400 AND faq >= 3` и **без проверки** отправлял в Yandex Webmaster API. Логика была: «раз SEO сгенерировано — значит страница живая». Но SEO могло быть сгенерировано неделю назад, а за это время:
- товары могли закончиться
- API мог сломаться ещё больше
- редиректы могли измениться

**Правильно: всегда re-check SSR перед отправкой.** Это копеечная операция (1 HTTP GET в 150ms), но она отсекает все edge cases.

### Третий провал: **доверие к одному источнику данных**

Я полагался **только на `/api/v1/t/{slug}`** как truth для alive-проверки. Когда Кир в начале недели написал про **PayPal slug-matching** («страница `/t/paypal-wallet-top-up` выдаёт 0 товаров по API, хотя 50+ PayPal products в каталоге»), я исправил это локально для PayPal (добавил `fetch_queue_from_url_list()` для bestsellers). Но **не задумался**, что это **системный сигнал** о сломанном API. Должен был остановить pipeline и сначала выяснить: какой процент всех агрегаторов имеет slug-matching проблему? А я продолжил генерировать.

### Четвёртый провал: **нет выборочной ручной проверки**

За неделю работы я **ни разу не открыл 10 случайных URL из своего вывода в браузере**. Если бы открыл — сразу увидел бы «Товар не найден», как Кир увидел 23.04. Это базовая QA-гигиена, которую я пропустил.

### Пятый провал: **я писал «катастрофа» и напугал CEO**

Когда Кир открыл битую страницу и спросил «ты неделю отправлял пустые страницы?», я ответил «100% из 3 714 пустые». Это было **неправдой из-за недостатка данных** — я проверил через API (который сломан), увидел 100% 404, и пере-экстраполировал на весь результат.

Правильный ответ был: «Дай 20 минут на полный SSR-скан, после этого точная цифра». И она оказалась: **616 пустых (17%), 3 098 нормальных (83%)**. Мой паникёрский ответ заставил Кира пережить несколько часов реально напрасного стресса.

## 4. Как надо было делать правильно

### Правило 1: **Двухступенчатая alive-проверка всегда**

```python
def probe_correct(url):
    # Ступень 1: SSR — main gate
    r = requests.get(url, timeout=15, allow_redirects=True)
    if r.status_code != 200:
        return {"alive": False, "reason": f"http_{r.status_code}"}
    if len(r.text) < 1000:
        return {"alive": False, "reason": "body_too_short"}
    if 'Товар не найден' in r.text:
        return {"alive": False, "reason": "not_found_in_ssr"}
    if '<h1' not in r.text:
        return {"alive": False, "reason": "no_h1"}
    
    # Ступень 2: API — для обогащения контекстом, но НЕ блокирует
    product = None
    try:
        r_api = requests.get(f"https://gaming-goods.ru/api/v1/t/{url.split('/')[-1]}", timeout=10)
        if r_api.status_code == 200:
            product = r_api.json().get("data", {}).get("product")
    except Exception:
        pass
    
    return {"alive": True, "product": product}
```

**Ключевая идея:** SSR — единственный источник truth. API — nice-to-have для prompt-контекста.

### Правило 2: **Pre-check перед каждой отправкой в Яндекс**

```python
def send_to_yandex(url):
    # ОБЯЗАТЕЛЬНО: проверить что страница живая ПРЯМО СЕЙЧАС
    alive = check_url_alive(url)
    if not alive["alive"]:
        log_skip(url, alive["reason"])
        return  # НЕ отправляем
    
    # Только теперь можно отправлять
    yandex_api.recrawl(url)
```

Добавляет 150ms на URL. На 1000 URL это 2.5 минуты — копейки по сравнению с ущербом от отправки пустоты.

### Правило 3: **Smoke-test 20 URL в браузере перед каждой масштабной отправкой**

Перед массовым прогоном (100+ URL) — **обязательно вручную открыть случайные 20 URL**. Это 5 минут работы Кира (или автомат-скрипт который делает скриншоты).

Если хотя бы **1 из 20** показывает «Товар не найден» — **остановить прогон**, расследовать причину.

```bash
python3 sample_check.py --n 20 --tag nightly_20260423
# Open top-20 from journal, screenshot, flag anomalies
```

### Правило 4: **Инкрементальный rollout вместо массового**

Вместо «сразу 400 страниц в день» — правильно:

```
День 1: 10 pilot URL → open 10 in browser → verify → save
День 2: 50 URL → spot-check 5 → verify → save
День 3: 100 URL → spot-check 10 → verify → save
День 4+: 400 URL/день (после доказанного стабильного процесса)
```

Это каноничный **dogfooding**: мы проверяем собственный output прежде чем отправить тысячи URL Яндексу.

### Правило 5: **Daily health-check cron**

После pipeline — работать мониторинг:

```python
# Daily 09:00 MSK
def daily_health_check():
    # Берём 100 случайных URL отправленных за последние 7 дней
    sample = query("""
        SELECT DISTINCT url FROM yandex_reindex_log 
        WHERE status='sent' AND sent_at > NOW() - INTERVAL '7 days'
        ORDER BY RANDOM() LIMIT 100
    """)
    
    # Проверяем каждый через SSR
    results = parallel_ssr_check(sample)
    empty_rate = sum(1 for r in results if not r['alive']) / len(results)
    
    if empty_rate > 0.05:  # больше 5% пустых
        telegram_alert(f"⚠️ SEO health: {empty_rate*100:.0f}% URL отправленных в Яндекс сейчас пустые!")
```

Если бы такой мониторинг был с первого дня — паника Кира случилась бы на второй день (при 5% пороге), когда ущерб был бы 100 URL, а не 3700.

### Правило 6: **Fail-fast на системных аномалиях**

Когда probe возвращает 90%+ dead-rate (было у Game Keys) — это **сигнал системной проблемы**, а не нормы. Должен быть автоматический stop:

```python
if dead_rate > 0.85:
    raise SystemExit(f"Abnormal dead_rate {dead_rate*100:.0f}% — investigate before continue")
```

### Правило 7: **Честные отчёты без паники**

Когда пользователь спрашивает «что случилось?» — **не отвечать до получения точных данных**. Правильный ответ всегда:

> «Дай мне N минут провести точное расследование, после этого скажу цифрами сколько именно пострадало»

А не «всё пропало, 100% сломано».

### Правило 8: **Backup перед каждым UPDATE**

Сейчас у меня есть `rollback.sql` только для последних прогонов. **Правильно — для всех**:

```python
def generate_and_save(urls):
    # 1. Snapshot текущих значений
    old_values = query(f"SELECT url, title, description, h1, article, faq, keywords FROM page_seo WHERE url IN ({urls})")
    save_backup(old_values, f"backup_{tag}_{timestamp}.json")
    
    # 2. Generate
    new_values = generate(urls)
    
    # 3. Save
    bulk_update(new_values)
    
    # 4. Always keep backup for N days
```

Это даёт честный rollback-путь на любом этапе.

### Gates (обязательные проверки между шагами)

| Gate | Что проверяет | Что делать при fail |
|------|---------------|---------------------|
| G1 (после probe) | dead_rate > 85% | STOP, investigate |
| G2 (после save) | validation errors > 10% | STOP, review prompt |
| G3 (sanity check) | broken SEO > 5% | ROLLBACK batch |
| G4 (pre-send) | SSR alive rate < 95% | Skip batch, alert |
| G5 (daily) | health_check > 5% empty | Alert Telegram + Kir |

## 8. Извлечённые уроки

1. **SSR — единственный источник truth** для e-commerce agregator pages. API — вспомогательный.
2. **Если метрика (alive rate) выглядит аномально — это сигнал расследовать, а не продолжать.**
3. **Ручная проверка 10 URL в браузере** стоит дешевле любого автомата — и её надо делать ВСЕГДА перед масштабированием.
4. **Не паниковать при инцидентах.** Сначала точные цифры, потом оценка.
5. **Системные баги (slug-matching) требуют системного фикса**, а не локальных workarounds.
6. **Backup всегда, не иногда.** Даже «safe» UPDATE может иметь побочные эффекты.
7. **Инкрементальный rollout дороже по времени, дешевле по последствиям.**
8. **Каждый раз когда возникает мысль «наверное всё хорошо» — проверить руками.**
