# 🚨 P0: API `/api/v1/t/{slug}` возвращает 404 для существующих агрегаторов

**Приоритет:** CRITICAL — P0
**Дата:** 23.04.2026
**Срочность:** сегодня-завтра максимум

---

## Что происходит

На прошлой неделе было сгенерировано SEO для 2 973 страниц и отправлено на recrawl 3 714 URL в Яндекс. При проверке 23.04 обнаружилось:

**100% проверенных URL в API `/api/v1/t/{slug}` возвращают 404** с ошибкой:
```json
{"error":{"code":"NOT_FOUND","message":"no products for brand: ea-star-wars-triple-bundle-accounts"}}
```

При этом:
- Страница в браузере отдаёт **HTTP 200**, но с блоком «Товар не найден»
- В БД `products` товары есть (по брендам EA, Electronic Arts — тысячи SKU)
- В БД `page_seo` SEO-контент качественный (title, description, article 800+ знаков, FAQ 6+)
- Для **части** URL agregator slug не совпадает с slug'ами товаров (тот же баг что с `paypal-wallet-top-up`)

**Пример проблемной страницы:** https://gaming-goods.ru/t/ea-star-wars-triple-bundle-accounts

---

## Что это значит для бизнеса

1. Яндекс индексирует пустые страницы → понижает их ранжирование → теряем органику
2. Пользователь кликает из выдачи → видит «Товар не найден» → уходит с сайта (bounce 100%)
3. Сам домен gaming-goods.ru начинает терять доверие поисковика (negative quality signals)

---

## Что нужно сделать (по важности)

### 1. Починить slug-matching в `/api/v1/t/{slug}`

Алгоритм сейчас (предположительно): ищет products с exact match по полю `brand` = slug. Это не работает потому что:
- Агрегатор slug `ea-star-wars-triple-bundle-accounts`
- Products имеют slug'и вроде `star-wars-battlefront-ii-account`, brand `EA`
- Точного матча `brand='ea-star-wars-triple-bundle-accounts'` нет в БД

**Нужен fuzzy matching:**
- Парсить slug агрегатора на токены: `ea`, `star`, `wars`, `triple`, `bundle`, `accounts`
- Искать products где `LOWER(brand) LIKE '%ea%'` + slug содержит `star-wars`
- Или использовать маппинг через отдельную таблицу `slug_aliases` / `aggregator_rules`

### 2. Пока API не починен — отдавать HTTP 410 Gone для «Товар не найден»

Критически важно чтобы **страница без товаров НЕ отдавала HTTP 200**. Правильный код — `HTTP 410 Gone` или `404 Not Found`. Это даст Яндексу чёткий сигнал не индексировать страницу.

**В SSR роутере Nuxt:**
```javascript
// на странице /t/{slug}
if (products.length === 0) {
  throw createError({ statusCode: 410, statusMessage: 'No products available' })
}
```

Это за 1 час работы решит 80% проблемы с Яндексом — пустые страницы вылетят из индекса сами через 1-2 обхода.

### 3. После фикса — пересобрать список живых агрегаторов

В БД есть ~60K записей в `page_seo` с `/t/` URL. Реально валидных (с товарами) — неизвестно сколько. Предположительно <15K.

**Нужен SQL-запрос:**
```sql
-- Агрегаторы с реальными товарами (после фикса slug-matching)
SELECT DISTINCT ps.url
FROM page_seo ps
WHERE ps.lang='ru'
  AND ps.url LIKE '%/t/%'
  AND EXISTS (
    SELECT 1 FROM products p 
    WHERE p.is_active=true 
      AND <ТВОЁ_НОВОЕ_УСЛОВИЕ_МАТЧИНГА>
  )
```

Это даст мне (Claude) корректный allow-list для будущих прогонов генерации.

### 4. Переотправить в Яндекс те что реально живые

После всех фиксов — когда список живых известен — повторно отправить их recrawl, чтобы Яндекс переобошёл и обновил индекс с нормальными товарами.
